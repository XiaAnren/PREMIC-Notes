值得关注的有两个脚本（普通用户无法访问第二个脚本）：
- `/public/home/premdev/datbin/ingest/get_ncepGFS.csh`负责下载标号分别为`GFS3`、`GFS4`、`GFS5`的数据：

- `/public/home/premdev/bin/put_gfs45comb.prem.csh`负责对上述三种数据进行压缩后处理并整合到`/public/home/premdev/data/datainput/gfs4/`，理想情况下会生成$00\text{--}06$的`GFS5`、$09\text{--}24$的`GFS4`、$27\text{--}84$的`GFS3`，同时这些数据都被过滤掉了不需要的变量，并将经纬度范围裁剪至经度$30^\circ\mathrm{E}\text{--}161^\circ\mathrm{E}$，纬度$0^\circ\mathrm{N}\text{--}60^\circ\mathrm{N}$。

很明显后处理脚本就是导致经纬度范围过小的罪魁祸首。

但为什么有的数据又是完整的呢？

### 进一步排查

`/public/home/premdev/data/datainput/gfs4/`中，`YYYYMMDD00_fh.00HH_tl.press_gr.0p5deg.grib2`的历史数据都被正常进行了后处理，但是`YYYYMMDD12_fh.00HH_tl.press_gr.0p5deg.grib2`却只有$27\text{--}84$的`GFS3`被后处理过，其余$00\text{--}24$都是原始的`GFS4`。

`crontab`里这两个脚本会每隔一小时就被调用一次，因此推测脚本成功执行了下载与后处理（`/public/home/premdev/logs/putgfs345.log`的日志文件可以作证，虽然普通用户同样无法访问），但是每天最后一轮运行时只下载了$00\text{--}24$的`GFS4`，没有进行后处理，因此导致原先的数据都被新下载的给覆盖掉了。

下载脚本负责下载五小时之前的数据：
```sh
set YMDH = `date +%Y%m%d%H -d "5 hour ago"`
set YMD = `echo $YMDH |cut -b 1-8`
set DD = `echo $YMDH |cut -b 7-8`
set HH = `echo $YMDH |cut -b 9-10`
```

后处理脚本负责处理四小时之前的数据：
```sh
set YMD1 = `date +%Y%m%d -d "36 hour ago"`
set YMD = `date +%Y%m%d -d "4 hour ago"`
set HH = `date +%H -d "4 hour ago"
```

这也就意味着，$04:00\text{--}05:00$之间启动的下载脚本会下载前一天$12$时起报的数据，而后处理脚本则会处理当日$00$时起报的数据。日志文件中处理$12$时起报数据与$00$时起报数据之间的`ls: No match.`就是最后的证据。

### 总结

后处理脚本导致`GFS`数据普遍被裁切过经纬度范围，但其与下载脚本中的处理时间不一致又导致有部分数据是正常的。

总之推荐优先使用$12$时起报，预报时效覆盖$00\text{--}21$小时的`GFS4`数据。
