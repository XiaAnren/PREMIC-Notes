## 20260126-XFS修复导致NFS句柄失效

### 问题描述

修好`admin`后普通用户登录计算节点后仍然无法正常加载`.bashrc`，同时`ls`报错：
```bash
>>> ls
    ls: cannot open directory .: Stale file handle
```
说明在NFS的服务端（`admin`）修复并重启后，客户端（计算节点）的缓存里还遗留着坏掉的句柄，因此报错“过期的文件句柄”。

### 未经验证的理想解决方案

直接在`iadmin`重启NFS服务：
```bash
>>> ssh iadmin
>>> systemctl restart nfs-server
```

### 试错过程

尝试把`/public`卸载再重新挂载：
```bash
>>> umount -l /public && mount /public
    mount: can't find /public in /etc/fstab
```

检查挂载配置：
```bash
>>> cat /etc/fstab 
    #
    # /etc/fstab
    # Created by anaconda on Tue Oct 27 21:28:23 2020
    #
    # Accessible filesystems, by reference, are maintained under '/dev/disk'
    # See man pages fstab(5), findfs(8), mount(8) and/or blkid(8) for more info
    #
    /dev/mapper/centos-root                   /     xfs  defaults 0 0
    UUID=5db545dc-3335-42d7-b67c-0a78c30af07b /boot xfs  defaults 0 0
    /dev/mapper/centos-swap                   swap  swap defaults 0 0
```
说明计算节点以前是通过其他方式挂载的，而非系统标准的静态挂载配置文件。

检查开机自启脚本：
```bash
>>> cat /etc/rc.local
    #!/bin/bash
    # THIS FILE IS ADDED FOR COMPATIBILITY PURPOSES
    #
    # It is highly advisable to create own systemd services or udev rules
    # to run scripts during boot instead of using this file.
    #
    # In contrast to previous versions due to parallel execution during boot
    # this script will NOT be run after all other services.
    #
    # Please note that you must run 'chmod +x /etc/rc.d/rc.local' to ensure
    # that this script will be executed during boot.

    touch /var/lock/subsys/local
    source /etc/nfs.local
>>> cat /etc/nfs.local 
    mkdir -p /public
    mount -t nfs -o vers=3 iadmin:/public /public
    mkdir -p /data
    mount -t nfs -o vers=3 istorage:/data /data
```

尝试按照脚本里的方式进行挂载：
```bash
>>> mount -t nfs -o vers=3 iadmin:/public /public
    mount.nfs: Stale file handle
```
说明内核的NFS客户端缓存里还记着那个坏掉的句柄，或者服务端还没完全释放刚才的会话。

在计算节点的种种尝试都不见效果，包括“强制卸载后清空内核缓存再挂载”、“重启NFS相关服务再挂载”、“销毁重建挂载点目录再挂载”：
```bash
>>> umount -f /public
    umount: /public: not mounted
>>> sync
>>> echo 3 > /proc/sys/vm/drop_caches
>>> mount -t nfs -o vers=3 iadmin:/public /public
    mount.nfs: Stale file handle
>>> 
>>> systemctl restart nfs-idmap
>>> systemctl restart rpcbind
>>> mount -t nfs -o vers=3 iadmin:/public /public
    mount.nfs: Stale file handle
>>> 
>>> rm -rf /public
>>> mkdir /public
>>> mount -t nfs -o vers=3 iadmin:/public /public
    mount.nfs: Stale file handle
```
那问题一定在服务端`iadmin`上。

服务端`iadmin`虽然修复了文件系统，但NFS服务进程`nfsd`可能还记着旧的文件句柄表或锁状态，需要彻底刷新一下，即在`iadmin`重启NFS服务：
```bash
>>> ssh iadmin
>>> systemctl restart nfs-server
```

再回到计算节点进行挂载（实际只在`node1`挂载了一下，到其他计算节点发现已经挂载上了，可能重启完NFS服务就全都恢复不需要手动挂载了）：
```bash
>>> ssh node1
>>> mount -t nfs -o vers=3 iadmin:/public /public
```
