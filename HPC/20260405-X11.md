## 20260405-配置计算节点X11图形界面转发

### 问题描述

计算节点（`node*`）不存在`/usr/bin/xauth`，因此无法像`GPU01`那样直接配置`~/.ssh/config`进行X11转发。

### 解决方案

从其他节点直接将`/usr/bin/xauth`复制到本地`$PATH`所包含的目录中。

### 问题描述

可以识别到`xauth`的可执行文件，但X11转发仍然不可用。

### 解决方案

`SSH`的X11转发机制是由服务端`sshd`进程在底层调用的，`sshd`会硬编码寻找系统标准路径（`/usr/bin/xauth`），普通用户修改环境变量无法改变`sshd`的调用行为。

<details>
<summary>经验证没有效果的方案</summary>

尝试绕过TCP端口，直接将底层的Socket文件转发至计算节点，从而省略鉴权环节：
```bash
ssh -R /tmp/.X11-unix/...
```
但是`/tmp/.X11-unix/`目录完全是空的，表明当前环境的X11连接基于纯TCP，根本不存在Socket文件可供转发。

尝试使用无鉴权的纯TCP端口转发：
```bash
ssh -R 6010:localhost:6010
```
尽管隧道能够成功建立，但X11协议除了要连端口还要比对秘钥，而当前没有提供有效的MIT-MAGIC-COOKIE，因此导致在计算节点上仍然`Can't open display`。

</details>

此前在测试`GPU01`等节点的过程中发现，必须使用`-Y`（受信任转发）而非`-X`（非受信转发）才能正常渲染现代GUI程序。说明只要能手动确保X11鉴权通过，且隧道本身没有安全扩展限制，图形界面即可正常工作。

因此尝试完全绕过`sshd`的原生X11转发机制，通过手动管理TCP隧道与鉴权密钥，等效实现`-Y`的效果：
```bash
_ssh_node() {
  local index=$(echo "$DISPLAY" | awk -F'[:.]' '{print $2}')
  local socket=$((6000 + index)) destination="localhost:$index.0"
  local cookie=$(xauth list "$DISPLAY" | awk '{print $3}')

  [[ -z "$index" || -z "$cookie" ]] && { ssh "$@"; return 0; }

  ssh -t -R "$socket:localhost:$socket" "$@" "export DISPLAY=$destination; xauth add $destination . $cookie; exec bash --login"
}
```

对于其中的`[[ -z "$index" || -z "$cookie" ]] && { ssh "$@"; return 0; }`：
- 经测试，在函数内使用`ssh "$@"`等效于直接在命令行使用`ssh`进行登录；
- 同时添加`return 0`防止退出登录后继续执行函数后续逻辑。

对于其中的`ssh -t -R "$socket:localhost:$socket" "$@" "export DISPLAY=$destination; xauth add $destination . $cookie; exec bash --login"`：
- 使用`-t`强制分配伪终端；
- 使用`-R`建立隧道；
- 登录后立即载入X11所必需的`$DISPLAY`和`$cookie`；
- 通过`exec bash --login`使用登录式Shell替换当前进程。

### 问题描述

上述登录方式会在后台进程列表中留下非常长的`ssh`命令记录。

### 解决方式

将相关命令写入临时文件，交由目标节点的`~/.bashrc`异步处理：
```bash
_ssh_node() {
  local index=$(echo "$DISPLAY" | awk -F'[:.]' '{print $2}')
  local socket=$((6000 + index)) destination="localhost:$index.0"
  local cookie=$(xauth list "$DISPLAY" | awk '{print $3}')

  [[ -z "$index" || -z "$cookie" ]] && { ssh "$@"; return 0; }

  echo "export DISPLAY=$destination; xauth add $destination . $cookie" > "$HOME/.SSH"; ssh -R "$socket:localhost:$socket" "$@"
}
```

同时在`~/.bashrc`中设置动态导入与销毁：
```bash
[[ -f "$HOME/.SSH" ]] && { source "$HOME/.SSH"; rm "$HOME/.SSH"; }
```

### 问题描述

使用`ssh16`登录到`node16`后，不论是先`ssh15`再`ssh16`登录回来，还是直接`ssh16`重复登录，都会因重复绑定端口导致冲突。

### 解决方案

添加针对“是否重复登录”与“目标端口是否正在被监听”的判断逻辑：
```bash
_ssh_node() {
  local index=$(echo "$DISPLAY" | awk -F'[:.]' '{print $2}')
  local socket=$((6000 + index)) destination="localhost:$index.0"
  local cookie=$(xauth list "$DISPLAY" | awk '{print $3}')

  [[ -z "$index" || -z "$cookie" ]] && { ssh "$@"; return 0; }

  if [[ "$(hostname -s)" != "$1" ]] && ! ssh -o ConnectTimeout=5 -o BatchMode=yes "$@" "/usr/sbin/ss -tln | grep -q ':$socket '"; then
    echo "export DISPLAY=$destination; xauth add $destination . $cookie" > "$HOME/.SSH"; ssh -R "$socket:localhost:$socket" "$@"
  else
    echo "export DISPLAY=$destination" > "$HOME/.SSH"; ssh "$@"
  fi
}
```

对于其中的`ssh -o ConnectTimeout=5 -o BatchMode=yes "$@" "/usr/sbin/ss -tln | grep -q ':$socket '"`：
- 使用`BatchMode=yes`确保探测端口时如果遇到密码输入或Host Key确认时不会卡住当前终端，而是直接失败返回；
- 非登录式Shell不会`source /etc/profile`，因此没有将`/usr/sbin`写入`$PATH`，必须使用绝对路径调用`ss`；
- 使用`grep -q ':$socket '`而非`grep -q ':$socket'`以避免`:6010`错误匹配到`:60100`的问题。

对于其中的`echo "export DISPLAY=$destination" > "$HOME/.SSH"; ssh "$@"`：
- 计算节点没有`/usr/bin/xauth`，不会自动分配`$DISPLAY`，所以仍然需要手动持久化当前的`$DISPLAY`；
- 首次跳转时已经通过`xauth add`将密钥写入了共享的`~/.Xauthority`，因此二次跳转时无需重复注入密钥。

### 问题描述

使用`ssh16`登录到`node16`后，再执行`ssh01`进行原生转发时，`node16`本地的`sshd`会硬编码寻找`/usr/bin/xauth`，但由于文件不存在所以无法自动分配`$DISPLAY`。

### 解决方案

在`~/.ssh/config`内为具备原生转发条件的节点显式指定`xauth`的路径（实际上规定的是登录发起方的`xauth`路径）：
```ssh-config
Host mgt login2 gpu01
  User USER
  ForwardX11 yes
  ForwardX11Trusted yes
  XauthLocation /public/home/XiaAnRen/share/bashrc/bin/xauth
```
