## 20260406-配置VSCode终端X11图形界面转发

### 问题描述

在原生支持X11转发的节点上使用VSCode终端时不会自动分配`$DISPLAY`，同时也无法打开X11图形窗口。

### 解决方案

在本地的`~/.ssh/config`中设置将远程的`60XX`端口（对应`DISPLAY=localhost:XX.0`）转发到本地的`6000`端口（对应`DISPLAY=localhost:0.0`，即`VcXsrv`一直在监听的端口）：
```ssh-config
Host login2 gpu01
  User USER
  ProxyJump mgt
  RemoteForward 60XX localhost:6000
```

同时在远程的`~/.bashrc`中手动指定与之对应的`$DISPLAY`：
```bash
[[ "$TERM_PROGRAM" == "vscode" && "$(hostname -s)" =~ ^(mgt|login2|gpu01)$ && -z "$DISPLAY" ]] && export DISPLAY=localhost:XX.0
```

需要注意的是，端口的选取仅限于`6010`-`6099`之间，同时要避开其他用户的已用端口，防止发生冲突。

### 问题描述

在计算节点上使用VSCode终端时不会自动分配`$DISPLAY`，同时也无法打开X11图形窗口。

### 解决方案

经测试没有想象中复杂，只要与其他节点一样在本地的`~/.ssh/config`中设置远程监听就能生效：
```ssh-config
Host node*
  User USER
  ProxyJump mgt
  RemoteForward 60XX localhost:6000
```

因此简化了远程的`~/.bashrc`（不再需要硬编码主机名作为判断条件）：
```bash
[[ "$TERM_PROGRAM" == "vscode" && -z "$DISPLAY" ]] && export DISPLAY=localhost:XX.0
```

### 问题描述

使用VSCode登录`node16`并打开终端后，走手动隧道的`ssh15`也能实现X11转发，但是走原生`ForwardX11`的`ssh01`又会警告`No xauth data; using fake authentication data for X11 forwarding.`。

### 解决方案

经测试尽管会出现警告，但实际还是可以继续使用X11转发功能的，因此考虑只在特殊情况下将警告过滤掉：
```bash
_ssh_x11() {
  if [[ "$(hostname -s)" =~ ^node && -n "$DISPLAY" && -z "$(xauth list "$DISPLAY")" ]]; then
    ssh "$@" 2> >(sed '/Warning: No xauth data; using fake authentication data for X11 forwarding./d' >&2)
  else
    ssh "$@"
  fi
}
```

> 还有一些边缘场景没有测试，但实际使用中很少涉及，当前方案已足够满足日常需求。
