## 20260126-断电导致XFS元数据损坏

### 问题描述

普通用户登录`admin`后无法正常加载`.bashrc`，同时`ls`报错：
```bash
>>> ls
    ls: cannot open directory .: Input/output error
```

### 问题排查

检查最近是否有重启：
```bash
>>> last -x | grep -B1 reboot
    runlevel (to lvl 5)  3.10.0-957.e17.x Mon Jan 26 00:41 - 18:10 (17:29)
    reboot   system boot 3.10.0-957.el7.x Mon Jan 26 00:40 - 18:10 (17:30)
```
00:40刚好是文德楼断电的时间点，而18:10则对应出现问题的时间点。

检查挂载情况：
```bash
>>> df -Th
    Filesystem Type Size Used Avail Use% Mounted on
    ...
    /dev/sdb1  xfs  99T  61T  39T   62%  /public
```
一切正常。

查看普通用户的家目录以及根目录情况：

>>> ll /

推测是底层的磁盘/文件系统出了严重问题（硬件故障或者XFS元数据损坏）。

进一步检查日志：
```bash
>>> journalctl -b -0 -p err..alert -n 200
    -- Logs begin at Sun 2026-01-25 16:40:31 GMT, end at Mon 2026-01-26 10:24:01 GMT. --
    ...
    Jan 26 07:55:36 admin kernel: XFS (sdb1): Internal error XFS_WANT_CORRUPTED_GOTO at line 1820 of file fs/xfs/libxfs/xfs_alloc.c.  Caller xfs_free_extent+0xaa/0x140 ...
    Jan 26 07:55:36 admin kernel: XFS (sdb1): Corruption of in-memory data detected.  Shutting down filesystem
    Jan 26 07:55:36 admin kernel: XFS (sdb1): Please umount the filesystem and rectify the problem(s)
    ...
```
说明XFS文件系统（`/dev/sdb1`）发生了元数据（metadata）破坏，内核主动把文件系统关掉了，之后访问它就会报`Input/output error`。

### 实际解决方案

安全重启再修复并挂载：
```bash
>>> grep public /etc/fstab
    # UUID=3cd9b78f-7284-4b92-bf21-41c80f4c9977 /public xfs defaults 0 0
>>> reboot
>>> xfs_repair -L /dev/sdb1
>>> mount /dev/sdb1 /public
>>> mount -t nfs -o vers=3 istorage:/data /data
```

### 包含试错过程的理想解决方案

先尝试卸载后修复：
```bash
>>> umount -l /public
>>> xfs_repair /dev/sdb1
    xfs_repair: cannot open /dev/sdb1: Device or resource busy
```
报错说明内核层面`/dev/sdb1`还被某个进程锁着，所以`xfs_repair`为了安全起见拒绝操作。

检查是哪个进程在占用：
```bash
>>> lsof /dev/sdb1
```
输出为空，说明内核自己因为之前的XFS Crash把锁给卡死了，只能进行安全重启再修复（这里是把`/public`挂载回去再执行以下操作的，但实际应该不需要）。

但是直接重启的话，系统启动时会尝试自动挂载`/public`，而因为它是坏的，系统可能会卡在启动界面进不去，或者再次把文件系统搞得更乱，所以得先修改`/etc/fstab`，把`/public`那一行注释掉，让开机不自动挂载`/public`：
```bash
>>> grep public /etc/fstab
    # UUID=3cd9b78f-7284-4b92-bf21-41c80f4c9977 /public xfs defaults 0 0
```

重启：
```bash
>>> reboot
```

重启后确认`/public`没有被挂载：
```bash
>>> df -h
    Filesystem Size  Used Avail Use% Mounted on
    /dev/sda3  259G  188G   71G  73% /
    devtmpfs   126G     0  126G   0% /dev
    tmpfs      126G     0  126G   0% /dev/shm

```

此外尝试挂载依然报错：
```bash
>>> mount /dev/sdb1 /public
    mount: mount /dev/sdb1 on /public failed: Structure needs cleaning
```

执行修复：
```bash
>>> xfs_repair /dev/sdb1
```

修复完成后恢复配置：
```bash
>>> grep public /etc/fstab
    UUID=3cd9b78f-7284-4b92-bf21-41c80f4c9977 /public xfs defaults 0 0
```

挂载（两种方式任选其一）：
```bash
>>> mount /public
>>> mount -a
```

后来有用户反馈缺少`/data`，因此根据[后续计算节点的经验](20260126-node.md)检查`admin`的开机自启脚本：
```bash
>>> cat /etc/rc.local
    #!/bin/bash
    ##!/bin/sh -e
    #
    # rc.local
    #
    # This script is executed at the end of each multiuser runlevel.
    # Make sure that the script will "exit 0" on success or any other
    # value on error.
    #
    # In order to enable or disable this script just change the execution
    # bits.
    #
    # By default this script does nothing.
    source /etc/nfs.local
    mount -t nfs -o vers=3 istorage:/data /data
    exit 0
>>> cat /etc/nfs.local
    mkdir -p /public
    mkdir -p /data
    mount -t nfs -o vers=3 istorage:/data /data
```

使用脚本里的方法进行挂载：
```bash
>>> mount -t nfs -o vers=3 istorage:/data /data
```


### 一些边角问题

为什么损坏了但没有立刻炸掉？

> 这种现象在存储领域被称为休眠错误或延时触发故障。
> 
> XFS作为高性能文件系统，采用了一种优化的挂载机制：启动时仅检查日志和超级块等核心结构，而非全盘扫描。如果核心元数据校验通过，XFS便会认为文件系统健康并直接挂载。
> 但实际上，断电已经导致某个分配组里的元数据逻辑错误。而这块元数据如果不被访问就不会造成影响。
> 直到故障触发时刻，系统进程需要进行空间分配或释放操作，内核访问到了该损坏的节点，发现校验和不匹配或指针异常，从而触发报错。日志里报错在`xfs_alloc.c`（空间分配模块）也印证了这一点：此前系统未对该分配区域进行操作，因此未触发故障。
> 
> 总结就是：硬件层面的数据损坏在断电时已经发生，但由于文件系统的懒加载特性，故障被延迟触发。当业务操作访问到损坏的元数据时，内核检测到一致性错误，为防止数据恶化，主动采取了文件系统保护性停机。
